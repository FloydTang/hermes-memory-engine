# Hermes Memory Engine — Architecture

## 概述

三层记忆架构借鉴自认知科学的工作记忆模型：
- **L0（自检层）** = 注意力监控
- **L1（活跃记忆）** = 工作记忆
- **L2（冷存储）** = 长时记忆

## 三层详解

### L0 — 自检层

**目标**：在零 LLM 成本下，维持记忆系统的健康水位。

**机制**：
1. 通过写入一条临时标记到记忆系统，从响应读取 `usage` 百分比
2. 判断阈值：
   - `<50%` → 健康，跳过
   - `50-70%` → 报告状态
   - `70-85%` → 生成压缩方案，等待确认
   - `>85%` → 需要立即压缩
3. 删除临时标记

**触发方式**：
- 每次 session 启动（agent 自行检查 system prompt 中的记忆内容）
- 周度 cron 定时触发

**成本**：每次检查消耗 2 次 memory API 调用（一次写 + 一次删），约 50ms。

---

### L1 — 活跃记忆

**目标**：高频使用的核心信息，保持在最快可达状态。

**载体**：系统内置 memory 工具，容量 2,200 chars。

**条目格式**：
```
[标签][优先级] 内容 — 关键细节
```

**优先级规则**：

| 优先级 | 含义 | 保留策略 | 示例标签 |
|--------|------|----------|----------|
| P0 | 核心不可丢失 | 永不归档永不删除 | `[行为]` `[核心]` `[身份]` |
| P1 | 重要但可归档 | 使用率>70%时可归档 | `[平台]` `[群规]` `[infra]` |
| P2 | 临时信息 | 完成后立即清除 | `[备忘]` `[待办]` |

**操作约束**：
- P0 条目任何时候不可删除或修改
- 压缩操作只影响 P1/P2
- 归档前先备份到 L2

---

### L2 — 冷存储

**目标**：完整保留历史数据，支持回溯恢复。

**载体**：文件系统中的 Markdown 文件。

**目录结构**：
```
~/.hermes/memory_archive/
├── INDEX.md                    ← 全局索引
├── 2026-05-19.md               ← 单次归档
└── YYYY-MM-DD.md               ← 更多归档
```

**INDEX.md 格式**：
```markdown
| 日期 | 文件 | 内容概要 |
|------|------|----------|
| 2026-05-19 | 2026-05-19.md | 描述 |
```

**恢复流程**：
1. 查询 INDEX.md 定位目标归档文件
2. 读取对应文件找到条目
3. 写入 memory 工具恢复

---

## 数据流

```
用户交互 → 产生需要记住的信息
              ↓
          L1 活跃记忆（memory 工具）
          ├─ P0 → 永久保留
          ├─ P1 → 使用率升高时 → L2 归档
          └─ P2 → 完成后 → 清除
              ↓
      L0 自检器（每周 cron）
      ├─ 检查使用率
      ├─ 压缩冗余 P1/P2
      └─ 检测重复模式
              ↓
          skill 候选 → hermes-auto-evolution-cron
```

## 与类似系统的对比

| 维度 | Hermes ME | mars-memory-engine | 传统 memory 方案 |
|------|-----------|---------------------|------------------|
| 层级 | 3层 | 3层（L0/L1/L2） | 通常 1 层 |
| 自检机制 | L0 主动检查 | L0 分类器 | 无 |
| 检索方式 | 全量注入 + 文件回溯 | 向量检索 + LangMem | 全量或 Key-Value |
| skill 回环 | ✅ 原生集成 | ❌ | ❌ |
| 实现复杂度 | 低（纯规则+标签） | 高（向量DB+分类器） | 低 |
